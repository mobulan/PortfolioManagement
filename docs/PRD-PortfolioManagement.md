# PRD：基估宝「个人资金管理看板 + 投资组合管理工具」

版本：v0.1
目标项目：`hzm0321/real-time-fund` 二次开发
目标用户：个人基金/ETF/现金/黄金资产管理者
核心目标：把现有“基金实时估值查看器”升级为“个人投资组合资产管理系统”。

### 2026-05-16 智能买卖执行口径更新

本轮新增“智能买卖”执行入口，作为 Excel/VBA 智能买卖体验在 Web 端的落地版本：

1. 用户只输入一个买入预算或卖出金额，系统实时判断本次应采用“按比例买卖”还是“填坑买入 / 削峰卖出”。
2. 判断阈值继续沿用 `thresholdAmount = totalAssets * theta`：输入金额绝对值大于阈值时按目标比例分配；小于等于阈值时按低配/超配资产优先处理。
3. 再平衡建议金额与可执行交易草稿分层：先按资产类别生成建议金额，再分配到具体持仓基金，并换算 `amount / priceEstimate = shareEstimate`。
4. 价格优先级为：基估宝今日预估价格 `gsz` / `estimatedNav`、最新单位净值 `dwjz` / `currentNav`、持仓估值、持仓净值；不得用成本价换算交易份额。交易草稿必须显示价格来源，缺少当日估值/单位净值时应阻断应用。
5. 卖出草稿不得超过当前可卖份额或可卖市值；超额卖出必须阻断应用并提示用户降低金额。
6. 预估价格只用于操作辅助。真实成交后仍应允许用成交净值和成交份额校正交易记录。

我先说明一下调研结论：原项目已经具备实时估值、重仓追踪、持仓管理、交易记录、定投计划、分组管理、明暗主题、导入导出和 Supabase 同步等基础能力，且采用 Next.js、JSONP/脚本注入直连金融数据源、本地 `localStorage` 持久化、可静态部署到 GitHub Pages。 项目知识库还显示当前代码是 Next.js 16 App Router、纯 JavaScript、`page.jsx` 较单体化，已有 shadcn/ui、Chart.js、Fuse.js、TanStack Query/Table、Zustand 等依赖或约定。 所以这次需求不应重写估值功能，而应在现有基金数据、交易记录、分组和同步体系之上新增“组合层”。

## 1. 产品定位

### 1.1 一句话描述

在基估宝现有实时基金估值能力之上，新增一个现代化的个人资金管理看板和投资组合管理系统，让用户能够统一管理自己持有的基金、股票类基金、债券类基金、黄金、现金等资产，并支持组合收益统计、细粒度交易记录、再平衡建议、历史导入导出和组合回测。

### 1.2 核心价值

当前 Excel 方案已经能做四大类资产：股票、债券、黄金、现金，并通过 VBA 实现按比例买入、智能填坑/削峰、强制再平衡、智能买卖和历史归档。 但它的限制是：

1. 只能较粗粒度地记录四大类资产。
2. 每类资产目前基本对应一只基金。
3. 手动更新历史和本金。
4. 无法直接调用基估宝的基金估值搜索、实时估值、交易记录、定投计划。
5. 回测、波动率、夏普比率、相关性等分析能力不足。

本次新增功能的目标是把 Excel 中的“永久投资组合记账逻辑”升级为 Web 端“多组合、多基金、多资产类别、可回测、可视化”的长期资金管理工具。

1. 不做真实交易下单。

5. 不优先改造原有基金估值主流程，避免影响现有用户使用。

#### E. 再平衡 Tab

新系统需要兼容这个格式，但不能停留在四大类聚合层。建议拆成两层：

1. `portfolioSnapshots`：组合每日总览快照。
2. `portfolioAssetSnapshots`：组合内资产类别每日快照。
3. `portfolioHoldingSnapshots`：组合内每只基金每日快照。

### 5.3.2 快照记录规则

| 防重复   | 同一组合同一日期默认只保留一条，可选择覆盖 |

### 5.3.4 历史曲线数据来源与节点推断

历史图表不应只依赖用户手动维护的每日快照。系统已经掌握基金代码、当前份额、成本、本金变化和交易记录时，应尽量自动还原可解释的历史曲线和买卖节点。

#### A. 曲线生成规则

1. 基金/ETF 持仓曲线：使用 `持有份额 × 历史净值/估值` 生成单只基金的历史市值曲线。
2. 资产类别曲线：同一资产类别下所有基金/ETF/现金/手动资产按日期汇总，生成股票、债券、黄金、现金等资产类别曲线。
3. 组合总资产曲线：按日期汇总所有资产类别市值，生成总资产走势。
4. 组合收益率曲线：优先使用快照中的总本金；没有快照本金时，使用导入本金变化、交易本金记录或当前成本近似计算。
5. 百分比堆叠图：每个日期按 `资产类别市值 / 当日组合总资产` 计算，展示资产类别历史占比。
6. 现金类资产：优先使用快照或交易流水中的现金余额；缺失时保持最近一次已知现金余额。

#### B. 买卖节点来源优先级

买入、卖出和资金变化节点应按可信度分层展示：

| 来源 | 说明 | 可信度 | 展示要求 |
| --- | --- | --- | --- |
| 显式交易记录 | 用户在买卖模块中记录的买入、卖出、转换、分红、现金流 | 高 | 直接标注为真实节点 |
| 导入历史表格 | Excel/CSV 中的本金变化、资产类别市值变化、当日收益变化 | 中 | 标注为导入推断节点 |
| 当前份额 + 成本推断 | 有当前份额和成本，但没有历史交易记录时，根据历史净值寻找可能买入日期 | 低 | 标注为系统推断节点，可编辑确认 |
| 无成本导入 | 只有当前份额，没有成本/本金时 | 低 | 默认视为今日买入，节点日期为导入日或当前日期 |

#### C. 成本反推购买节点

当导入持仓有当前份额和成本金额，但缺少历史购买记录时：

1. 计算平均成本净值：`成本金额 / 当前份额`。
2. 拉取该基金历史净值。
3. 寻找最接近平均成本净值的历史日期，作为可能买入日期。
4. 如果存在多个接近日期，优先选择距离当前日期更近、且能与资产类别本金变化匹配的日期。
5. 生成一个“推断买入”节点，金额为成本金额，份额为当前份额，价格为推断成本净值。
6. 推断节点不得覆盖真实交易记录，必须允许用户后续确认、修改或删除。

该规则只能还原“可能的平均买入位置”，不能精确还原多次定投、分红再投、拆分、手续费、卖出后再买入等复杂路径。因此 UI 必须区分“真实节点”和“推断节点”。

#### D. 基于历史表格和本金变化的节点推断

当导入 Excel/CSV 历史表格时：

1. 如果某日总本金增加，默认推断为资金转入或买入节点。
2. 如果某日总本金减少，默认推断为资金转出或卖出节点。
3. 如果某一资产类别本金变化明显，优先将节点归属到该资产类别。
4. 如果同时存在买卖模块交易记录，以显式交易记录为准；历史表格仅补足缺失日期和缺失资产类别。
5. 如果本金没有变化但资产类别市值变化，视为市场波动，不推断买卖节点。
6. 如果本金变化无法归属到具体基金，只生成资产类别级节点，不强行绑定基金。

#### E. 图表节点展示

1. 总资产走势、资产类别曲线和单基金曲线都应支持节点标注。
2. 买入节点使用国内金融习惯的强调色，卖出节点使用相反色；推断节点使用空心或虚线样式。
3. Hover 节点时展示日期、类型、金额、份额、价格、来源、可信度。
4. 图例允许开关：真实交易、导入推断、成本推断、分红/现金流。
5. 当节点过密时，移动端默认聚合，点击后展开列表。

### 5.3.5 历史图表 UI 布局

图表交互风格应与基估宝现有基金曲线保持一致，但组合图表不直接复用单基金 `FundTrendChart` 业务组件。应复用 Chart.js 技术栈、主题色、红涨绿跌、时间范围按钮、Hover 十字线、空状态和移动端触摸体验，同时新建组合专用图表组件。

推荐布局：

1. 概览 Tab：放轻量趋势摘要，例如总资产走势小图和当前资产占比。
2. 历史 Tab：作为完整图表区。
3. 第一行：总资产走势折线图、总收益率走势折线图；PC 并排，移动端上下排列。
4. 第二行：资产类别百分比堆叠图，整行展示。
5. 第三行：每日盈亏柱状图、回撤曲线、本金变化阶梯图，可折叠展示。
6. 右侧栏继续承载记录快照、导入导出、CSV/Excel 操作，不放核心趋势图。
7. 少于 2 条快照或缺少历史净值时，展示明确空状态：提示用户记录更多快照、导入历史表格或补充基金代码。

## 8. 核心计算规则

### 8.1 当前市值

```
currentValue = share * estimatedNav
```

现金类：

```
currentValue = manualValue ?? amount
```

### 8.2 累计收益

```
profit = currentValue - costAmount
returnRate = costAmount === 0 ? 0 : profit / costAmount
```

### 8.3 当日预估收益

优先使用估值净值：

```
dailyEstimatedProfit = share * (estimatedNav - previousNav)
```

如果无法获得前一净值：

```
dailyEstimatedProfit = currentValue * estimatedGrowthRate
```

### 8.4 组合偏离度

```
currentRatio = assetClassValue / portfolioTotalValue
drift = currentRatio - targetRatio
```

### 8.5 总偏离度 Theta

沿用 Excel VBA 逻辑：

```
theta = sum(abs(currentRatio[i] - targetRatio[i]))
thresholdAmount = totalAssets * theta
```

当新增/取出资金金额大于阈值时，按目标比例买卖；小于等于阈值时，执行填坑/削峰。

### 8.6 再平衡建议金额

```
targetAmount = totalAssets * targetRatio
rebalanceAmount = targetAmount - currentAmount
```

正数表示买入，负数表示卖出。

在基估宝现有实时基金估值能力之上，新增一个现代化的个人资金管理看板和投资组合管理系统，让用户能够统一管理自己持有的基金、股票类基金、债券类基金、黄金、现金等资产，并支持组合收益统计、细粒度交易记录、再平衡建议、历史导入导出和组合回测。

### 1.2 核心价值

当前 Excel 方案已经能做四大类资产：股票、债券、黄金、现金，并通过 VBA 实现按比例买入、智能填坑/削峰、强制再平衡、智能买卖和历史归档。 但它的限制是：

1. 只能较粗粒度地记录四大类资产。
2. 每类资产目前基本对应一只基金。
3. 手动更新历史和本金。
4. 无法直接调用基估宝的基金估值搜索、实时估值、交易记录、定投计划。
5. 回测、波动率、夏普比率、相关性等分析能力不足。

本次新增功能的目标是把 Excel 中的“永久投资组合记账逻辑”升级为 Web 端“多组合、多基金、多资产类别、可回测、可视化”的长期资金管理工具。

1. 不做真实交易下单。
2. 不承诺估值绝对准确，仍沿用公开接口数据，仅作为个人记录与参考。
3. 不做券商/基金平台账户自动同步，首版以手动记录、导入、估值联动为主。

5. 不优先改造原有基金估值主流程，避免影响现有用户使用。

### 5.3.4 历史曲线数据来源与节点推断

历史图表不应只依赖用户手动维护的每日快照。系统已经掌握基金代码、当前份额、成本、本金变化和交易记录时，应尽量自动还原可解释的历史曲线和买卖节点。

#### A. 曲线生成规则

1. 基金/ETF 持仓曲线：使用 `持有份额 × 历史净值/估值` 生成单只基金的历史市值曲线。
2. 资产类别曲线：同一资产类别下所有基金/ETF/现金/手动资产按日期汇总，生成股票、债券、黄金、现金等资产类别曲线。
3. 组合总资产曲线：按日期汇总所有资产类别市值，生成总资产走势。
4. 组合收益率曲线：优先使用快照中的总本金；没有快照本金时，使用导入本金变化、交易本金记录或当前成本近似计算。
5. 百分比堆叠图：每个日期按 `资产类别市值 / 当日组合总资产` 计算，展示资产类别历史占比。
6. 现金类资产：优先使用快照或交易流水中的现金余额；缺失时保持最近一次已知现金余额。

#### B. 买卖节点来源优先级

买入、卖出和资金变化节点应按可信度分层展示：

| 来源 | 说明 | 可信度 | 展示要求 |
| --- | --- | --- | --- |
| 显式交易记录 | 用户在买卖模块中记录的买入、卖出、转换、分红、现金流 | 高 | 直接标注为真实节点 |
| 导入历史表格 | Excel/CSV 中的本金变化、资产类别市值变化、当日收益变化 | 中 | 标注为导入推断节点 |
| 当前份额 + 成本推断 | 有当前份额和成本，但没有历史交易记录时，根据历史净值寻找可能买入日期 | 低 | 标注为系统推断节点，可编辑确认 |
| 无成本导入 | 只有当前份额，没有成本/本金时 | 低 | 默认视为今日买入，节点日期为导入日或当前日期 |

#### C. 成本反推购买节点

当导入持仓有当前份额和成本金额，但缺少历史购买记录时：

1. 计算平均成本净值：`成本金额 / 当前份额`。
2. 拉取该基金历史净值。
3. 寻找最接近平均成本净值的历史日期，作为可能买入日期。
4. 如果存在多个接近日期，优先选择距离当前日期更近、且能与资产类别本金变化匹配的日期。
5. 生成一个“推断买入”节点，金额为成本金额，份额为当前份额，价格为推断成本净值。
6. 推断节点不得覆盖真实交易记录，必须允许用户后续确认、修改或删除。

该规则只能还原“可能的平均买入位置”，不能精确还原多次定投、分红再投、拆分、手续费、卖出后再买入等复杂路径。因此 UI 必须区分“真实节点”和“推断节点”。

#### D. 基于历史表格和本金变化的节点推断

当导入 Excel/CSV 历史表格时：

1. 如果某日总本金增加，默认推断为资金转入或买入节点。
2. 如果某日总本金减少，默认推断为资金转出或卖出节点。
3. 如果某一资产类别本金变化明显，优先将节点归属到该资产类别。
4. 如果同时存在买卖模块交易记录，以显式交易记录为准；历史表格仅补足缺失日期和缺失资产类别。
5. 如果本金没有变化但资产类别市值变化，视为市场波动，不推断买卖节点。
6. 如果本金变化无法归属到具体基金，只生成资产类别级节点，不强行绑定基金。

#### E. 图表节点展示

1. 总资产走势、资产类别曲线和单基金曲线都应支持节点标注。
2. 买入节点使用国内金融习惯的强调色，卖出节点使用相反色；推断节点使用空心或虚线样式。
3. Hover 节点时展示日期、类型、金额、份额、价格、来源、可信度。
4. 图例允许开关：真实交易、导入推断、成本推断、分红/现金流。
5. 当节点过密时，移动端默认聚合，点击后展开列表。

### 5.3.5 历史图表 UI 布局

图表交互风格应与基估宝现有基金曲线保持一致，但组合图表不直接复用单基金 `FundTrendChart` 业务组件。应复用 Chart.js 技术栈、主题色、红涨绿跌、时间范围按钮、Hover 十字线、空状态和移动端触摸体验，同时新建组合专用图表组件。

推荐布局：

1. 概览 Tab：放轻量趋势摘要，例如总资产走势小图和当前资产占比。
2. 历史 Tab：作为完整图表区。
3. 第一行：总资产走势折线图、总收益率走势折线图；PC 并排，移动端上下排列。
4. 第二行：资产类别百分比堆叠图，整行展示。
5. 第三行：每日盈亏柱状图、回撤曲线、本金变化阶梯图，可折叠展示。
6. 右侧栏继续承载记录快照、导入导出、CSV/Excel 操作，不放核心趋势图。
7. 少于 2 条快照或缺少历史净值时，展示明确空状态：提示用户记录更多快照、导入历史表格或补充基金代码。

## 8. 核心计算规则

### 8.1 当前市值

```
currentValue = share * estimatedNav
```

现金类：

```
currentValue = manualValue ?? amount
```

### 8.2 累计收益

```
profit = currentValue - costAmount
returnRate = costAmount === 0 ? 0 : profit / costAmount
```

### 8.3 当日预估收益

优先使用估值净值：

```
dailyEstimatedProfit = share * (estimatedNav - previousNav)
```

如果无法获得前一净值：

```
dailyEstimatedProfit = currentValue * estimatedGrowthRate
```

### 8.4 组合偏离度

```
currentRatio = assetClassValue / portfolioTotalValue
drift = currentRatio - targetRatio
```

### 8.5 总偏离度 Theta

沿用 Excel VBA 逻辑：

```
theta = sum(abs(currentRatio[i] - targetRatio[i]))
thresholdAmount = totalAssets * theta
```

当新增/取出资金金额大于阈值时，按目标比例买卖；小于等于阈值时，执行填坑/削峰。

### 8.6 再平衡建议金额

```
targetAmount = totalAssets * targetRatio
rebalanceAmount = targetAmount - currentAmount
```

正数表示买入，负数表示卖出。

1. 基金净值异常波动提醒。

可以给每个组合一个“健康度”：

```

显示：

```
优秀 / 正常 / 需关注 / 需要再平衡
```

### 12.4 目标规划

1. 设置目标金额，例如 50 万。
2. 设置目标日期，例如 2030 年。
3. 根据当前本金、定投计划、假设收益率计算达成概率。
4. 显示资金缺口。

1. 月度报告。

例如：

```

```

首版可以不做真实平台同步，但可以让用户手动标记资产来源。

### 12.7 税费与手续费

1. 买入费率。
2. 卖出费率。
3. 持有天数相关费率。
4. 管理费不单独处理，但可以备注。

支持：

1. 平均成本法。
2. FIFO。
3. 手动成本。
4. 仅历史记录，不影响当前持仓。

现金不是简单的“第四类资产”，应支持：

1. 真实现金。

3. 逆回购。

```
为什么建议买入债券 1200 元？

# 23. 最终推荐执行路线

最稳妥的路线是：

```
第 1 步：clone 原项目，创建 PortfolioManagement 仓库
第 2 步：设置 origin/upstream
第 3 步：建立 portfolio 独立目录和页面入口
第 4 步：做数据模型和计算引擎
第 5 步：做 Dashboard 和组合管理 UI
第 6 步：迁移交易、本金、快照
第 7 步：导入你的 CSV 历史数据
第 8 步：加入回测
第 9 步：建立 upstream 同步流程
第 10 步：持续从 real-time-fund 合并更新
```

核心策略一句话：

```
PortfolioManagement 不应该变成 real-time-fund 的硬分叉，而应该是“保留上游能力 + 新增 portfolio 模块”的增强版。
```

这样未来原项目更新估值接口、基金搜索、UI 修复、数据同步能力时，你仍然可以持续合并，而自己的组合管理功能也不会每次都被冲突拖垮。
